Comprehensive Final Exam Study Notes (English)

Part 1: SDLC (Software Development Life Cycle)

The entire process of a software system from inception to retirement.

Correct Order (Memorize):
1. Planning: Why build this system? (Problem Definition).
2. Analysis: WHAT will the system do? (Requirements) - Problem Domain
3. Design: HOW will it work? (Blueprints/Architecture) - Solution Domain
4. Implementation: Coding.
5. Testing: Finding bugs and errors.
6. Maintenance: Updates and support after release.
Critical distinction for the exam:
Analysis: Focuses on the Problem Domain. (Requirements, What).
Design: Focuses on the Solution Domain. (Technical Specs, How).

Part 2: Feasibility Analysis (Chapter 6)

Deciding whether to proceed with the project. The decision is "GO" or "NO GO".

The 6 Feasibility Tests

  • 1. Operational: Does it solve the user's problem? Will it be used? (PIECES framework is used here).
  • 2. Cultural: Will the organizational culture accept this system?
  • 3. Technical: Do we have the technology and expertise?
  • 4. Schedule: Is the deadline realistic?
  • 5. Legal: Is it compliant with laws and copyrights?
  • 6. Economic: The most important one. Cost-Benefit Analysis.

Economic Metrics

Term Definition
ROI Return on Investment. A high percentage is good.
NPV Net Present Value. The value of future money today. Must be positive (+).
Break-even Point The point in time where the project covers its costs and starts making a profit.

Part 3: Software Design (Chapter 5)

The heart of the course. The Golden Rule: High Cohesion and Low Coupling.

A. Cohesion (Module Strength)

How focused a module is on a single task. From Best (Green) to Worst (Red):

  • 1. Functional (Best): Module performs exactly one task.
    Ex: CalculateNetPay().
  • 2. Sequential: Output of one task is input for the next.
  • 3. Communicational: Tasks use the same data/input.
  • 4. Procedural: Tasks follow a specific execution order but don't share data.
  • 5. Temporal: Tasks must happen at the same "time".
    Ex: Init() or startup functions.
  • 6. Logical: Logically similar tasks grouped together (bad practice).
    Ex: A huge function handling Mouse, Keyboard, and Disk inputs together.
  • 7. Coincidental (Worst): Random, unrelated tasks in one module.

B. Coupling (Dependency)

How dependent two modules are on each other. From Best (Green) to Worst (Red):

  • 1. Data (Best): Only simple data parameters are passed (int, string).
  • 2. Stamp: Data structures (Struct/Object) are passed.
    Risk: The module has access to more data than it needs.
  • 3. Control (Exam Topic): One module sends a "Flag" to control the logic of another.
    Ex: Passing true to tell the function what to do.
  • 4. Common: Using Global Variables. (Very bad for maintenance).
  • 5. Content (Worst): One module directly modifies the internal code/data of another.

Part 4: Software Architectures

The strategic blueprints for the system.

  • Layered: Separates UI, Logic, and Data. Standard for Enterprise apps. Easy to maintain.
  • Repository: Centralized Database with independent modules around it. Used when Data Consistency is key.
  • Event-Driven: System reacts to messages/events. Good for Asynchronous and Scalable systems.
  • Pipe & Filter: Data flows through a pipeline. Input -> Process -> Output. (Unix, Compilers).
  • Client-Server: Workload distributed between requester (Client) and provider (Server).

Part 5: Design Patterns

You must know which pattern belongs to which category.

1. Creational

Deals with object creation mechanisms.

  • Singleton: Ensures a class has ONLY ONE instance. (e.g., DB Connection).
  • Factory Method: Lets subclasses decide which object to create.
  • Abstract Factory: Creates FAMILIES of related objects (e.g., MacButton + MacWindow).
  • Builder: Constructs complex objects step-by-step.
  • Prototype: Creates new objects by cloning existing ones.

2. Structural

Deals with object composition and structure.

  • Adapter: Makes incompatible interfaces work together. (Like a power plug adapter).
  • Facade: Provides a simplified interface to a complex system. (Like a car ignition key).
  • Decorator: Adds new functionality/behavior dynamically.
  • Proxy: Represents another object to control access (The "Surrogate").

3. Behavioral

Deals with communication between objects.

  • Observer: Notifies subscribers when an object changes. (YouTube Notification).
  • Strategy: Makes algorithms interchangeable. (Navigation routes).
  • Command: Turns a request into an object. (Supports Undo/Redo).
  • State: Object alters behavior when its internal state changes.
  • Iterator: Traverses a collection without exposing its structure.
Comparison Question:
What is the difference between Adapter and Facade?
Answer: Adapter fixes incompatibility between interfaces. Facade simplifies a complex interface.
